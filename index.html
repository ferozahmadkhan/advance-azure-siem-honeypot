Why Cowrie?

The industry standard SSH/Telnet honeypot — trusted by security researchers worldwide

01
🎭
Realistic Simulation

Cowrie emulates a complete Linux environment. Attackers cannot tell it is fake — they see realistic file systems, commands, and responses.

02
🪤
Intentionally Accepts Logins

Unlike real SSH, Cowrie lets everyone in on purpose. Attackers think they have access while every command they run is recorded in detail.

03
🔒
Safe by Design

Runs as an unprivileged user. Attackers cannot escape the fake shell to reach the real system. Complete isolation guaranteed.

04
📊
Structured JSON Logging

Every keystroke, command, and download attempt is logged in JSON format — perfect for shipping to Azure Log Analytics and querying with KQL.

💡 The Movie Set Analogy

A burglar breaks into what they think is a real house — but it's a Hollywood set with fake furniture. Cameras record everything while the burglar finds nothing of value. This is exactly what Cowrie does — every attacker thinks they won, while we watch and record.

Setup in 10 Steps

Complete guide — from zero to live honeypot on Azure Free Trial

01
Create Resource Group

Create a container for all resources. This makes cleanup easy — delete the group and everything goes.

Azure Portal → Resource Groups → + Create Name: honeypot-rg | Region: East US
02
Deploy Free Ubuntu VM

Deploy a B1s or B2ats_v2 VM (free eligible). Download the .pem key when prompted — save it safely.

Image: Ubuntu Server 22.04 LTS Size: Standard_B2ats_v2 (free eligible) Auth: SSH public key
03
Configure NSG Rules

Open ports for attackers and a secret admin port. Never use port 22 for real admin access once Cowrie is running.

Port 22 → Allow (honeypot trap) Port 23 → Allow (telnet trap) Port 4444 → Allow (your admin door)
04
Install Cowrie

Create a dedicated unprivileged user, download Cowrie, set up Python virtual environment.

sudo adduser --disabled-password cowrie sudo git clone https://github.com/cowrie/cowrie.git /home/cowrie/cowrie sudo chown -R cowrie:cowrie /home/cowrie/cowrie sudo su - cowrie && cd cowrie python3 -m venv cowrie-env && source cowrie-env/bin/activate pip install -r requirements.txt
05
Configure Cowrie Ports

Set Cowrie to listen on 2222 and 2223 (not 22/23 — Linux blocks privileged ports for non-root users).

cp etc/cowrie.cfg.dist etc/cowrie.cfg # [ssh] section: listen_endpoints = tcp:2222:interface=0.0.0.0 # [telnet] section: enabled = true # listen_endpoints = tcp:2223:interface=0.0.0.0
06
Set Up iptables Redirect

Redirect public port 22 to Cowrie's internal port 2222. Save rules permanently so they survive reboots.

sudo iptables -t nat -A PREROUTING -p tcp --dport 22 -j REDIRECT --to-port 2222 sudo iptables -t nat -A PREROUTING -p tcp --dport 23 -j REDIRECT --to-port 2223 sudo apt-get install -y iptables-persistent && sudo netfilter-persistent save
07
Start Cowrie

Launch the honeypot. Watch live logs to confirm attackers are being captured.

twistd --pidfile var/run/cowrie.pid -l var/log/cowrie/cowrie.log cowrie tail -f var/log/cowrie/cowrie.log
08
Create Log Analytics + Sentinel

Set up the Azure SIEM stack. Sentinel gives you 31 days free — more than enough to build this.

Azure Portal → Log Analytics Workspaces → Create Name: law-honeypot | Region: match your VM Azure Portal → Microsoft Sentinel → Add → Select workspace
09
Configure Log Pipeline

Create a Data Collection Rule to ship Cowrie's JSON logs to Log Analytics automatically.

Sentinel → Content Hub → Custom Logs via AMA → Install Data Connectors → Custom Logs via AMA → Create DCR File pattern: /home/cowrie/cowrie/var/log/cowrie/cowrie.json* Table name: CowrieLogs_CL Record delimiter: End of Line
10
Create KQL Detection Rules

Write analytics rules that automatically detect and alert on attack patterns.

// Brute Force Detection CowrieLogs_CL | where RawData contains "cowrie.login.failed" | extend src_ip = extract('"src_ip":"([^"]+)"', 1, RawData) | summarize FailedAttempts = count() by src_ip, bin(TimeGenerated, 5m) | where FailedAttempts > 2
